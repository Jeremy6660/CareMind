# Git 安装与使用指南（新手保姆级 · VS Code 图形化版）

Git 是目前最流行的版本控制系统。简单来说，它能帮你**记录代码的每一次修改**，方便回退、协作和备份。

本指南**以 VS Code 图形化操作为主**，配合 Git Graph 插件可视化分支历史，大幅降低学习成本——大部分日常操作点按钮就能完成，无需记忆命令。

---

## 一、Git 是什么？

打个比方：写论文时，你可能会存 `论文_v1.docx`、`论文_v2.docx`、`论文_最终版.docx`、`论文_最终版2.docx`……Git 就是帮你自动做这件事的工具，而且做得更好——它能记录**谁、什么时间、改了什么**，还能随时回到任意一个历史版本。

---

## 二、安装 Git

### 2.1 Windows

1. 打开浏览器，访问 [https://git-scm.com/download/win](https://git-scm.com/download/win)
2. 下载会自动开始，如果没有，点击页面上的 "Click here to download"
3. 双击下载好的 `.exe` 安装包
4. 安装过程中，**一路点 "Next" 即可**，所有选项保持默认
5. 安装完成后，在桌面空白处右键，菜单里会出现 **"Git Bash Here"**，说明安装成功

### 2.2 macOS

**方法一：直接安装（推荐新手）**

1. 打开浏览器，访问 [https://git-scm.com/download/mac](https://git-scm.com/download/mac)
2. 下载 `.dmg` 文件，双击打开
3. 双击 `.pkg` 文件，按照提示完成安装

**方法二：用 Homebrew（如果你已经装了 Homebrew）**

```bash
brew install git
```

### 2.3 验证安装

打开终端（Windows 打开 Git Bash，macOS 打开终端），输入：

```bash
git --version
```

如果看到类似 `git version 2.45.0` 的输出，说明安装成功。

## 三、安装 VS Code 及必备插件

### 3.1 安装 VS Code

如果还没装 VS Code，去 [https://code.visualstudio.com/](https://code.visualstudio.com/) 下载安装。

### 3.2 安装 Git Graph 插件（核心）

Git Graph 能以图形化方式展示分支、提交历史，让 Git 仓库结构一目了然。

1. 打开 VS Code，点击左侧活动栏的**扩展图标**（四个方块）
2. 搜索 **"Git Graph"**
3. 找到作者为 `mhutchie` 的插件，点击 **安装**
4. 安装完成后，VS Code 底部状态栏会出现一个 **"Git Graph"** 按钮，点击即可打开

> 你也可以用快捷键 `Cmd+Shift+P`（macOS）或 `Ctrl+Shift+P`（Windows）打开命令面板，输入 `Git Graph: View Git Graph` 打开。

### 3.3 推荐辅助插件

| 插件名 | 作用 |
|--------|------|
| **GitLens** | 增强 VS Code 内置 Git 能力，显示每行代码的修改者和修改时间 |
| **Git History** | 查看单个文件的修改历史 |

以上两个非必需，但装了会更好用。

## 四、初次配置（必须做）

安装完 Git 后，需要告诉 Git 你是谁。打开 VS Code 终端（`Ctrl+`` 或菜单栏 → 终端 → 新建终端`），依次输入：

```bash
git config --global user.name "你的名字"
git config --global user.email "你的邮箱@example.com"
```

例如：

```bash
git config --global user.name "张三"
git config --global user.email "zhangsan@gmail.com"
```

这只需要配置一次，以后所有项目都会自动使用这个身份。

## 五、核心概念（5 分钟搞懂）

| 概念 | 通俗解释 |
|------|----------|
| **仓库（Repository）** | 一个被 Git 管理的项目文件夹，所有版本历史都藏在里面的 `.git` 文件夹中 |
| **工作区** | 你正在编辑的文件夹，能看到、能改的那些文件 |
| **暂存区** | 一个"购物车"，你决定哪些修改要列入这次记录 |
| **提交（Commit）** | 一次"快照"——把暂存区的内容永久保存到历史记录中 |
| **分支（Branch）** | 一条独立的开发线，可以理解成"平行宇宙"，在分支上随便改也不会影响主线 |
| **远程仓库（Remote）** | 存放在 GitHub / GitLab 等服务器上的仓库副本 |
| **推送（Push）** | 把本地的修改上传到远程仓库 |
| **拉取（Pull）** | 把远程仓库的最新修改下载到本地 |

## 六、VS Code 图形化日常操作（主力）

VS Code 左侧活动栏有一个**源代码管理图标**（树枝分叉图案，快捷键 `Ctrl+Shift+G`），这是所有 Git 操作的入口。

### 6.1 打开项目 & 识别仓库

用 VS Code 打开任意项目文件夹后，如果该文件夹是一个 Git 仓库（包含 `.git` 文件夹），左侧源代码管理面板会自动识别并显示所有修改。

### 6.2 查看修改

当你编辑文件后，VS Code 会做两件事：

- **文件后面的标记**：在文件资源管理器中，文件名右边会出现字母标记——
  - **U**（Untracked）= 新文件，Git 还没跟踪
  - **M**（Modified）= 已修改
  - **A**（Added）= 已暂存待提交
- **行号旁边的色条**：编辑器里，行号左边会显示**绿色**（新增行）、**蓝色**（修改行）、**红色**（删除行）的指示条，一目了然

点击源代码管理面板里的任意文件，可以展开看到**修改前后的对比视图**（diff 视图）。

### 6.3 暂存（Add）→ 提交（Commit）→ 推送（Push）

这是你每天用到最多的操作，全程不需要终端：

1. **暂存**：在源代码管理面板中，把鼠标悬停在"更改"分组下的某个文件上，点击旁边的 **+** 按钮，该文件就进入了暂存区。如果要暂存所有文件，点"更改"标题行旁边的 **+** 即可。

2. **提交**：在面板顶部的输入框中写好提交信息（比如"修复了登录页面的 bug"），然后按 `Ctrl+Enter`（macOS: `Cmd+Enter`）或点击上方的 **✓ 提交** 按钮。

3. **推送**：提交后，VS Code 底部状态栏左侧会出现一个**同步图标**（循环箭头），旁边显示待推送的提交数量。点击它即可推送到远程仓库。

   > 也可以点状态栏左下角的**分支名**，在弹出菜单中选择"推送"。

**一句话总结：改代码 → 点 + 暂存 → 写信息按 Ctrl+Enter 提交 → 点状态栏同步按钮推送。**

### 6.4 拉取（Pull）

每天开工前先拉取最新代码，避免冲突：

- **方法一**：点击 VS Code 底部状态栏左下角的**同步图标**
- **方法二**：按 `Cmd+Shift+P` / `Ctrl+Shift+P` 打开命令面板，输入 `Git: Pull` 回车
- **方法三**：打开 Git Graph 面板，点击顶部的**向下箭头（Pull）**按钮

**好习惯：每天开工前先 Pull，收工前先 Push。**

### 6.5 分支操作

**查看 / 切换分支**：点击 VS Code 底部状态栏左下角的分支名（比如 `main`），弹出分支列表，点击即可切换。

**创建新分支**：同样点击状态栏分支名，在弹出菜单中选择 **"创建新分支..."**，输入分支名即可。

**合并分支**：
1. 先切换到目标分支（比如要合并到 `main`，就切到 `main`）
2. 按 `Cmd+Shift+P` → 输入 `Git: Merge Branch` → 选择源分支

> 以上操作在 Git Graph 中更直观（见下一章）。

### 6.6 解决合并冲突

当两人修改了同一个文件的同一行时，合并会产生冲突。VS Code 的冲突处理非常友好：

1. 冲突文件在源代码管理面板会被标记为 **C**（Conflict）
2. 点击该文件，编辑器中会显示冲突区域，格式为：
   ```
   <<<<<<< HEAD         （你当前分支的内容）
   你的修改
   =======              （分隔线）
   别人的修改
   >>>>>>> other-branch （对方分支的内容）
   ```
3. VS Code 会在冲突区域上方给出几个按钮：
   - **Accept Current Change** → 保留你的版本
   - **Accept Incoming Change** → 用别人的版本
   - **Accept Both Changes** → 两个都要
   - **Compare Changes** → 并排对比
4. 选择后删除多余标记，保存文件，然后暂存 → 提交这个解决了冲突的文件即可。

## 七、Git Graph 可视化操作（进阶利器）

Git Graph 把抽象的分支和历史变成直观的图形，强烈推荐每个新手都装上。

### 7.1 打开 Git Graph

点击 VS Code 底部状态栏的 **Git Graph** 按钮，或按 `Cmd+Shift+P` 输入 `Git Graph: View Git Graph`。

你会看到一颗"树"——每个圆圈是一个提交，从上到下按时间排列；不同颜色的线代表不同分支；分支名是彩色标签，贴在对应的提交上。

### 7.2 Git Graph 能做什么

在任意一个提交圆圈上**右键**，可以：

| 操作 | 说明 |
|------|------|
| **Checkout Branch** | 切换到这个提交（只读，不会丢失任何东西） |
| **Create Branch** | 从这个提交创建新分支 |
| **Cherrypick** | 把该提交的修改"复制"到当前分支（脚本的提交历史补丁） |
| **Revert** | 撤销该提交（生成一个新提交来反转修改，安全） |
| **Reset → Hard** | 回退到这里，丢弃之后的所有提交（危险操作，仅限自己的分支） |
| **Merge** | 把该分支合并到当前分支 |
| **Tag** | 给这个提交打个标签（比如 `v1.0`） |

在左侧的**分支列表**上右键，可以：
- 切换分支、重命名分支、删除分支、推送分支

面板顶部的按钮栏：
- **Pull**（向下箭头）：拉取远程代码
- **Push**（向上箭头）：推送本地提交
- **Fetch**（圆圈）：从远程获取最新信息但不合并

### 7.3 典型场景：用 Git Graph 理解分支与合并

1. 打开 Git Graph，观察当前分支结构
2. 在任意提交上右键 → **Create Branch**，起个名比如 `feature-test`
3. 现在你就在新分支上，做点修改、提交
4. 切回 `main` 分支（双击左侧分支列表中的 `main`）
5. 在 `feature-test` 分支的最新提交上右键 → **Merge into current branch**
6. Git Graph 中会看到一条新的合并线——直观展示合并历史

## 八、命令行速查（偶尔会用到）

大部分操作 VS Code 图形界面已经够用，但以下命令有时在终端里更快：

```bash
# ===== 查看状态 =====
git status                 # 查看哪些文件被修改了
git log --oneline          # 简短的提交历史（等效于看 Git Graph 但更快）

# ===== 暂存和提交 =====
git add .                  # 暂存所有修改
git commit -m "提交信息"    # 提交

# ===== 分支 =====
git switch -c 新分支名      # 创建并切换到新分支
git merge 分支名            # 合并分支

# ===== 远程操作 =====
git push                   # 推送
git pull                   # 拉取
git clone 仓库地址          # 克隆远程仓库到本地

# ===== 回退（用得少，优先用 Git Graph 右键 Revert）=====
git stash                  # 暂存当前修改，让工作区变干净
git stash pop              # 恢复刚才暂存的修改
```

## 九、写提交信息的好习惯

- **用动词开头**："添加登录功能" 而不是 "登录功能"
- **简单描述做了什么**，不需要写怎么做的（代码已经说明了）
- **一次提交只做一件事**：不要 "修复 bug + 添加新功能 + 改样式" 混在一起

> 不好的：`修改`
>
> 好的：`修复登录页密码输入框无法粘贴的问题`

## 十、常见问题

### Q1：提交信息写错了怎么办？

在 VS Code 中：源代码管理面板 → 点击右上角 `...` → `Commit` → `Commit Staged (Amend)`，修改提交信息后重新提交。

> 只有最新一次提交还没 Push 时才能这样改。

### Q2：提交了不该提交的文件怎么办？

先创建 `.gitignore` 文件（在 VS Code 资源管理器右键 → 新建文件），把不想跟踪的文件名或文件夹名写进去，每行一个：

```
node_modules/
.env
*.log
.DS_Store
```

如果已经提交了 `node_modules`，去终端执行：

```bash
git rm -r --cached node_modules
git commit -m "从仓库中移除 node_modules"
```

### Q3：Push 时提示 "rejected" 怎么办？

说明别人已经推送了新代码，你需要先拉取再推送：点击状态栏同步按钮先 Pull，如有冲突按第六章方法解决，然后再 Push。

### Q4：怎么撤销已经 Push 的提交？

用 Git Graph 最方便：在要撤销的那个提交上右键 → **Revert**，然后推送。这会生成一个新的"反向提交"，安全且不破坏历史。

### Q5：不小心改乱了，想回到上次提交的状态

如果你只是某个文件改乱了：源代码管理面板 → 在该文件上右键 → **放弃更改**。

如果想全部恢复：用 Git Graph，在当前分支的最新提交上右键 → **Reset → Hard**（注意：后续修改会丢失，仅限在自己分支上操作）。

## 十一、建议学习路径

1. **第一天**：安装 Git → 配置用户名邮箱 → 在 VS Code 中学会暂存 + 提交
2. **第二天**：注册 GitHub → 学会克隆、推送、拉取（点按钮就行）
3. **第三天**：装上 Git Graph → 创建分支 → 合并分支 → 通过可视化理解分支概念
4. **需要时再看**：Revert、Reset、Stash 等高级操作（Git Graph 右键菜单已经帮你包好了）

用 VS Code + Git Graph 的组合，你不需要记住任何命令就能完成 90% 的日常操作。建议从今天开始把自己的项目用 Git 管理起来，同时在 Git Graph 中观察仓库的变化，很快就能建立直觉。

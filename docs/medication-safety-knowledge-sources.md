# CareMind 用药安全知识库来源整理

> 适用模块：Agent 2 用药安全  
> 目标用途：为后续 `medication_conflicts.json`、用药风险提示规则、测试用例设计提供权威来源索引与录入规范。  
> 整理日期：2026-05-27

## 一、建库原则与安全边界

CareMind 的用药安全知识库只用于 **照护辅助、风险提醒和沟通支持**，不能替代医生、药师或医疗机构的诊断、处方、停药、换药决策。

Agent 2 的输出应始终使用“提示”“建议咨询医生/药师”“建议复核”这类表达，避免使用“必须停药”“应立即改药”等超出系统职责的措辞。

知识库进入系统前需要满足三条底线：

1. **来源可追溯**：每条规则必须能追溯到指南、标准、药品说明书、监管数据库或经同行评议的数据源。
2. **风险可解释**：不仅记录“有冲突”，还要记录风险机制、可能后果、风险等级和适用人群。
3. **人工复核**：所有涉及用药禁忌、剂量调整、替代药建议的内容，正式使用前必须由医学或药学背景人员复核。

## 二、来源分级

| 等级 | 来源类型 | 用途 | 入库优先级 |
|------|----------|------|------------|
| A | 老年人潜在不适当用药标准、国家/国际指南 | 建立核心风险规则 | 最高 |
| A | 药品说明书、监管机构药物警戒信息 | 校验禁忌、警告、特殊人群用药 | 最高 |
| B | 结构化药物相互作用数据库 | 批量发现药物组合风险 | 高 |
| B | 长期照护/高警示药物清单 | 标记养老场景重点关注药物 | 高 |
| C | 综述、论文、案例报告 | 补充背景和机制解释 | 中 |
| D | 网络文章、非官方科普、无来源表格 | 仅作线索，不可直接入库 | 不建议 |

## 四、初期采集重点

第一版 `medication_conflicts.json` 不建议追求“大而全”，应优先覆盖养老机构最常见、风险最高、最容易被护工忽略的药物类别。

| 类别 | 重点药物示例 | 主要风险 |
|------|--------------|----------|
| 抗血小板/抗凝药 | 阿司匹林、氯吡格雷、华法林、利伐沙班 | 出血风险、跌倒后严重后果、与 NSAIDs 叠加风险 |
| NSAIDs | 布洛芬、双氯芬酸、塞来昔布 | 消化道出血、肾功能损害、血压控制变差 |
| 降压药 | 氨氯地平、硝苯地平、卡托普利、缬沙坦 | 低血压、跌倒、肾功能相关风险 |
| 降糖药 | 二甲双胍、格列齐特、胰岛素 | 低血糖、肾功能相关风险 |
| 镇静催眠药 | 地西泮、阿普唑仑、佐匹克隆、唑吡坦 | 谵妄、跌倒、认知功能下降 |
| 抗胆碱药 | 苯海索、部分抗过敏药、部分三环类抗抑郁药 | 便秘、尿潴留、认知恶化、谵妄 |
| 心脏相关药物 | 地高辛、胺碘酮、利尿剂 | 心律失常、电解质紊乱、肾功能相关风险 |
| 胃药/PPI | 奥美拉唑、埃索美拉唑 | 长期使用风险、与部分药物相互作用 |
| 阿片类镇痛药 | 吗啡、羟考酮、曲马多 | 呼吸抑制、跌倒、便秘、镇静叠加 |
| 锂盐等窄治疗窗药物 | 碳酸锂 | 中毒风险、肾功能和相互作用敏感 |

## 五、建议知识库字段模板

后续建立 JSON 数据时，建议每条规则采用如下字段：

```json
{
  "id": "ddi_aspirin_ibuprofen_001",
  "drug_a": "阿司匹林",
  "drug_b": "布洛芬",
  "drug_a_aliases": ["aspirin"],
  "drug_b_aliases": ["ibuprofen"],
  "risk_level": "high",
  "risk_type": "bleeding_risk",
  "mechanism": "两者均可能增加胃肠道出血风险，老年人风险更高",
  "effect": "消化道出血风险增加；合并抗凝药或既往消化道出血史时需高度警惕",
  "evidence_level": "B",
  "suggestion": "提示照护人员联系医生或药师复核；不要自行停药或换药",
  "applicable_population": ["older_adults", "long_term_care"],
  "source": [
    {
      "name": "AGS Beers Criteria 2023",
      "url": "https://pmc.ncbi.nlm.nih.gov/articles/PMC12478568/"
    }
  ],
  "review_status": "pharmacist_review_required",
  "last_reviewed": "2026-05-27"
}
```

字段说明：

| 字段 | 含义 |
|------|------|
| `id` | 唯一规则编号，建议使用药物组合和序号 |
| `drug_a` / `drug_b` | 中文通用名，优先使用国内常用名称 |
| `drug_a_aliases` / `drug_b_aliases` | 英文名、商品名、常见别名 |
| `risk_level` | 建议使用 `low`、`medium`、`high`、`critical` |
| `risk_type` | 风险类型，如出血、低血糖、跌倒、谵妄、肾损害 |
| `mechanism` | 简短说明为什么有风险 |
| `effect` | 可能造成的照护后果 |
| `evidence_level` | 证据强度，可先用 A/B/C 或“指南/标签/数据库/案例” |
| `suggestion` | 面向护工的安全表达，必须避免替代处方决策 |
| `source` | 来源名称和链接，允许多个 |
| `review_status` | 审核状态，如 `draft`、`pharmacist_review_required`、`approved` |
| `last_reviewed` | 人工复核日期 |

## 六、审核与更新流程

### 1. 资料摘录

由资料整理人员从上述来源中摘录候选规则。摘录时必须保留：

- 原始来源名称；
- 原始链接；
- 药物名称；
- 风险描述；
- 适用条件；
- 是否涉及老年人、肾功能、认知障碍、跌倒、出血等养老高频风险。

### 2. 药学复核

正式进入 Agent 2 前，每条规则至少需要一次药学复核。复核重点包括：

- 药物名称是否准确；
- 风险是否适用于中国养老场景；
- 建议措辞是否越权；
- 是否存在过度报警；
- 是否遗漏关键适用条件。

### 3. 测试用例覆盖

每条高风险规则都应至少配一个测试用例：

- 输入：老人画像 + 当前用药清单；
- 预期输出：是否触发风险；
- 预期风险等级；
- 预期提示语；
- 不应触发的相似药物组合。

Agent 2 的测试原则是：**宁可保守提醒，也不能漏掉明确高风险组合**。但过多误报会降低护工信任度，因此必须记录误报原因并持续优化。

### 4. 定期更新

建议更新节奏：

| 内容 | 更新频率 |
|------|----------|
| 国内 PIM 标准、老年医学指南 | 每 6-12 个月检查一次 |
| 药品说明书和监管安全通报 | 每 3-6 个月检查一次 |
| DDI 数据库 | 每 6 个月检查一次 |
| 项目内 JSON 规则 | 每次新增演示案例或测试失败后更新 |

### 5. 输出措辞规范

Agent 2 面向护工和家属时，应使用以下安全表达：

- “检测到可能存在用药风险，建议联系医生或药师复核。”
- “该组合在老年人中需要特别注意出血/跌倒/低血糖风险。”
- “请勿自行停药、减药或换药。”
- “如出现黑便、明显头晕、意识改变、低血糖症状等，请及时联系医护人员。”

避免使用以下表达：

- “必须停用某药。”
- “应改用某药。”
- “系统诊断为药物中毒。”
- “该药绝对不能使用。”

## 七、后续工作建议

下一步可以围绕首批 50 条用药风险规则开展：

1. 从中国 PIM 标准、Beers、STOPP/START 中提取老年高风险药物。
2. 用 DDInter、DailyMed、openFDA 校验药物相互作用和说明书警告。
3. 将高频养老药物优先做成 JSON。
4. 每条规则配一个测试病例。
5. 请药学或医学背景人员完成第一轮复核。

推荐第一批先做 20 条高风险组合，跑通 Agent 2 的检索、匹配、提示生成和人工复核流程，再扩展到 50 条。

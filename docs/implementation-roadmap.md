# 实现路线图：CareMind 三月执行计划

> **从设计到答辩：2026年6月-9月的里程碑与交付物**

---

## 总体时间规划

| 月份 | 阶段 | 核心目标 | 交付物 |
|------|------|---------|--------|
| **6月** | 设计期 | 需求锁定、架构确认、知识库筹建 | 需求规格书、架构设计文档、知识库初版 |
| **7月** | 开发期 | 单Agent开发、核心模块联调 | 三个Agent原型、用户画像模块 |
| **8月** | 集成期 | 多Agent集成、前端Demo、系统联调 | 完整系统原型、集成测试报告 |
| **9月** | 冲刺期 | 测试打磨、文档完善、答辩准备 | 研究报告、演示视频、答辩PPT |

## 6月：设计期

### 6月上旬（1-15日）：需求分析与架构设计最终确认

**任务**：
- [ ] 针对三大Agent的需求分析
  - Agent 1：分诊评估的输入/输出规格、风险等级定义、触发逻辑
  - Agent 2：药物禁忌库的数据结构、预警等级设计、假正率目标
  - Agent 3：报告生成的模板库、个性化维度的优先级排序
- [ ] 用户画像模块的需求锁定
  - 认知评级标准（正常/轻度/中重度）
  - 方言选项与TTS支持的方言清单
  - 关系身份分类（老人本人/护工/家属）
- [ ] Orchestrator的协同逻辑设计
  - 共享记忆结构设计
  - Agent间冲突仲裁规则
  - 错误处理与降级方案

**输出**：
- 需求规格书（需求分析文档）
- 最终架构设计文档（包括数据流图、交互序列图）

### 6月中旬（15-22日）：RAG知识库筹建

**知识库来源准备**：

| 来源 | 具体内容 | 获取方式 | 优先级 |
|------|---------|---------|--------|
| 中国老年医学临床指南 | 常见老年病诊疗规范（冠心病、糖尿病、骨关节炎等） | 中华医学会官网 / 医学数据库 | 🔴 最高 |
| 多药并用禁忌数据库 | NSAIDs相互作用、降压药+利尿剂组合、抗凝血药相互作用等 | 参考美国Beers标准 + 中国本地化指南 | 🔴 最高 |
| 长期护理保险政策 | 失能等级评估标准、保险范围、理赔流程 | 国家医保部门官文 | 🟡 高 |
| 机构照护规范 | 养老机构管理规范、照护操作标准、卫生防疫要求 | 民政部行业标准 | 🟡 中 |

**任务**：
- [ ] 文献收集与整理（建立本地知识库）
- [ ] 数据结构设计（向量数据库schema设计）
- [ ] RAG索引建立（使用LlamaIndex或同类工具）

**输出**：
- RAG知识库初版（可在向量数据库中检索）
- 知识库质量评估报告

### 6月下旬（22-30日）：开发环境搭建与技术预研

**任务**：
- [ ] 开发环境准备
  - Python环境（版本≥3.9）
  - 依赖库安装（LangGraph / AutoGen、LlamaIndex、FastAPI等）
- [ ] 技术选型确认
  - LLM API选择（Claude / GPT-4o / Qwen）
  - 向量数据库选择（Pinecone / Weaviate / Milvus）
  - 前端框架初选（React / Vue 或微信小程序框架）
- [ ] 开发分支与CI/CD初设（如有）

**输出**：
- 技术预研报告
- 开发环境Checklist（新开发者按这个快速上手）

## 7月：开发期

### 7月上旬（1-10日）：Agent 1 分诊评估开发

**核心逻辑**：
```
输入：症状描述 + 历史记录 + 基线数据
↓
[大模型分析]
↓
输出：风险等级 + 处置建议 + 触发其他Agent的条件
```

**任务**：
- [ ] 分诊评估prompt设计与测试
  - 设计系统prompt（角色定位、评估标准、输出格式）
  - 设计few-shot示例（正常/轻度/中度/重度的典型案例）
  - 测试误报率与漏报率
- [ ] 风险等级定义与触发规则
  - 绿色（正常）：症状正常，定期监测
  - 黄色（轻度异常）：需要关注，触发Agent 3通知家属
  - 红色（高危）：需要医学干预，触发Agent 2 + 紧急通知
- [ ] 个性化交互逻辑
  - 认知正常：开放式对话
  - 轻度障碍：是/否式简化选项
  - 重度障碍：护工代理输入

**单元测试**：
- 编写10-15个测试用例（覆盖常见老年病症状）
- 验证准确率、灵敏度、特异度

**输出**：
- Agent 1原型代码
- 分诊逻辑详细设计文档
- 单元测试报告

### 7月中旬（10-20日）：Agent 2 用药安全开发

**核心逻辑**：
```
输入：老人当前用药清单
↓
[对比禁忌库 + 多药协同分析]
↓
输出：风险预警 + 建议措施
```

**任务**：
- [ ] 多药并用禁忌知识库构建
  - 常见老年病药物对及其协同效应（NSAIDs、降压药、抗凝血药等）
  - 关联风险等级与证据等级
  - **数据质量至关重要**（涉及生命安全，必须用官方指南）
- [ ] 冲突检测算法设计
  - 两两对比检测
  - 多药协同风险检测（如 A+B 安全，但 A+B+C 不安全）
- [ ] 预警信息生成
  - 生成个性化的预警（考虑老人年龄、肝肾功能等）
  - 生成家属通知草稿

**单元测试**：
- 编写15-20个测试用例（覆盖高频药物组合）
- 验证误报率与漏报率
- **注意**：宁可多报（假正），不要漏报（假负）

**输出**：
- Agent 2原型代码
- 多药禁忌数据库与检测算法文档
- 单元测试报告

### 7月下旬（20-31日）：Agent 3 家属沟通 & 用户画像模块

**Agent 3 的核心逻辑**：
```
输入：老人的日常照护数据 + 健康指标 + 交互记录
↓
[根据家属沟通偏好 + 关系身份生成个性化报告]
↓
输出：温暖的、故事化的照护报告
```

**任务**：
- [ ] 照护报告生成的模板库设计
  - 模板1：常规周报（正常状态）
  - 模板2：预警报告（出现异常时）
  - 模板3：康复进展报告（恢复期）
  - 每个模板支持多个个性化变量
- [ ] 个性化维度的实现
  - 语言复杂度调整（根据认知评级）
  - 方言适配（未来集成TTS）
  - 关系视角（同一信息的3个版本：老人/护工/家属）
- [ ] 用户画像模块开发
  - 认知评级数据结构（问卷或简易评估）
  - 方言偏好与TTS映射表
  - 关系图谱存储（老人-家属-照护者的关系定义）

**单元测试**：
- 生成10-15份样本报告
- 进行易读性评测（是否简洁、温暖、有信息量）

**输出**：
- Agent 3原型代码
- 用户画像模块代码与数据结构设计
- 样本报告与易读性评测

## 8月：集成期

### 8月上旬（1-10日）：多Agent集成与Orchestrator开发

**核心任务**：将三个独立的Agent整合成一个协同系统

**Orchestrator 的职责**：
- 接收用户输入
- 调用相应的Agent（或多个Agent）
- 管理Agent间的共享记忆（如"用户的当前风险等级是什么"）
- 处理冲突（如Agent 1说"黄色"，Agent 2说"红色"时，谁的判断优先？）

**任务**：
- [ ] Orchestrator架构设计
  - 基于LangGraph或AutoGen的Agent工作流设计
  - 共享记忆结构（Store老人的当前状态、风险等级、过往交互历史）
  - 冲突仲裁规则（定义Agent间优先级）
- [ ] API接口设计
  - 输入接口（老人症状、用药清单等）
  - 输出接口（分诊结果、用药预警、家属报告）
  - 数据格式统一（JSON）
- [ ] 端到端集成测试
  - 设计3-5个完整场景
  - 验证三个Agent的协同效果

**输出**：
- Orchestrator核心代码
- API接口文档
- 集成测试报告

### 8月中旬（10-20日）：前端Demo与个性化生成模块联调

**任务**：
- [ ] 简易前端原型（Web或微信小程序）
  - 老人/护工的症状输入界面
  - 系统输出的展示界面
  - 家属的报告查看界面
- [ ] 个性化生成模块的联调
  - 测试不同认知等级的输出是否差异化
  - 测试不同方言下的TTS（如果已集成）
  - 测试三角色视角的内容差异
- [ ] 用户交互流程验证
  - 一个完整的从"输入症状"到"生成报告"的端到端流程

**输出**：
- 前端原型代码（可运行的Demo）
- 个性化生成的验证报告

### 8月下旬（20-31日）：系统联调与性能优化

**任务**：
- [ ] 全链路联调与缺陷修复
  - 发现并修复Agent间的接口问题
  - 优化大模型的调用延迟（缩短响应时间）
- [ ] 知识库与Agent的匹配测试
  - Agent 2 与多药禁忌库的检索效果
  - RAG的准确率与召回率评测
- [ ] 性能与稳定性测试
  - 并发处理能力（同时处理多个老人的请求）
  - 错误处理（网络故障、API超时等）

**输出**：
- 完整系统原型（可演示）
- 系统测试报告（性能数据、缺陷清单）

## 风险与缓解方案

| 风险 | 影响 | 缓解方案 |
|------|------|---------|
| 知识库质量不足 | Agent 2的预警准确率低，降低系统可信度 | 提前与医学专家合作验证；宁可保守（少报），不要冒生命风险 |
| 大模型API成本超支 | 预算爆炸 | 提前测试API调用频率，考虑本地部署Qwen |
| 用户测试反馈不好 | 需要大幅改进，时间紧张 | 提前做可用性测试（8月中旬），预留改进时间 |
| 技术选型变更 | 需要重写代码 | 谨慎选择技术栈，6月中旬前冻结选择 |
| 团队成员离职或生病 | 工期延误 | 确保知识共享（代码注释、设计文档清晰） |

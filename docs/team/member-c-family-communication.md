# 成员 C 分工安排：Agent 3 家属沟通负责人

## 一、团队总分工

CareMind 采用“四条主线、互相验收”的小组协作方式。每个人负责一个清晰模块，同时承担对其他成员产出的审查责任，避免最后集成时才发现接口、内容或安全边界不一致。

| 成员 | 角色定位 | 主责模块 | 核心交付物 | 评价指标 |
|---|---|---|---|---|
| A | 分诊评估负责人 | Agent 1 | 症状案例库、分诊 Prompt、分诊函数、测试记录 | 风险分级准确率、建议是否安全清晰 |
| B | 用药安全负责人 | Agent 2 | 药物风险 JSON、用药匹配逻辑、预警 Prompt、来源审核表 | 高风险组合不漏报、来源可追溯 |
| C | 家属沟通负责人 | Agent 3 | 老人画像、报告模板、沟通 Prompt、三方报告生成 | 表达温暖、信息准确、不掩盖风险 |
| D | 集成与交付负责人 | Orchestrator + Demo + 材料 | 项目骨架、Gradio Demo、端到端流程、报告/PPT 汇总 | Demo 可演示、接口统一、材料完整 |

互审机制：

- A 审 C：确认家属报告里的健康风险没有被“温暖表达”稀释。
- B 审 A：确认分诊建议没有越权诊断或医学风险。
- C 审 B：确认用药预警能被护工和家属读懂。
- D 审全员：确认接口、文件结构、演示流程可以集成。

## 二、C 的角色定位

C 是 Agent 3 家属沟通与个性化表达负责人，负责把冷冰冰的照护数据转化为家属、护工和老人都能理解的表达。

Agent 3 是 CareMind 最能体现“全人照护”和“老人主体性”的模块。它的目标不是美化数据，而是在准确传递健康信息的同时，让家属看到老人作为一个具体的人，而不是一组指标。

## 三、阶段任务

| 阶段 | 时间 | 任务重点 | 产出 |
|---|---|---|---|
| 准备期 | 6 月 | 准备老人画像、报告模板和沟通 Prompt | `elder_profiles.json`、5 组报告对比、Prompt 初版 |
| 开发期 | 7 月 | 开发三方报告生成函数 | `generate_family_report()`、报告模板 |
| 集成期 | 8 月 | 参与 Demo 试用反馈，优化报告表达 | 外部试用反馈、报告修订版 |
| 冲刺期 | 9 月 | 支持报告和 PPT 中个性化生成部分 | 个性化机制说明、展示样例 |

## 四、具体措施

1. 建立至少 3 个老人画像，覆盖认知正常、轻度认知障碍、中重度认知障碍。
2. 每个老人画像至少包含：姓名、年龄、认知水平、方言或语言偏好、疾病、用药、过敏史、性格、爱好、家庭联系人。
3. 准备 5 组“传统版 vs CareMind 温暖版”报告对比。
4. 5 组报告建议覆盖：常规周报、有异常需关注、康复进展、节日或生日、家属久未探视。
5. 设计家属沟通 Prompt，支持老人版、护工版、家属版三种输出。
6. 开发函数 `generate_family_report(elder_id, weekly_data) -> dict`。
7. 接收 Agent 1 的分诊结果和 Agent 2 的用药预警，并把关键风险写入报告。

建议老人画像结构：

```json
{
  "id": "elder_001",
  "name": "张奶奶",
  "age": 77,
  "cognitive_level": "normal",
  "dialect": "普通话",
  "conditions": ["高血压", "骨关节炎"],
  "medications": ["氨氯地平 5mg qd", "阿司匹林 100mg qd"],
  "allergies": ["青霉素"],
  "personality": "开朗健谈，喜欢回忆年轻时的事，怕孤独",
  "hobbies": ["唱歌", "散步", "红烧肉"],
  "family_contacts": [
    {
      "relation": "儿子",
      "name": "王先生",
      "prefer_weekly_report": true
    }
  ]
}
```

建议输出结构：

```json
{
  "elder_version": "张奶奶，2026-06-08 状态不错，我们下午一起散了步。",
  "caregiver_version": "今日血压略高，建议继续记录并观察头晕情况。",
  "family_version": "张奶奶本周精神不错，但 2026-06-08 出现轻微头晕，我们已记录并建议继续观察血压变化。",
  "risk_items": ["今日头晕需观察", "用药组合需复核"],
  "source_facts": ["血压 142/88", "下午散步 15 分钟"]
}
```

## 五、工作要求

- 报告要温暖，但不能隐藏异常指标、分诊风险和用药风险。
- 家属版避免医学黑话，重点解释“发生了什么、机构做了什么、家属需要知道什么”。
- 护工版强调行动建议，例如继续记录、观察时长、联系医生或药师复核。
- 老人版语言更短、更温和，适合认知能力下降的老人理解。
- 个性化必须基于画像字段，不凭空编造老人经历、家庭关系或爱好。
- 不能把家属久未探视写成责备语气，要以支持和邀请为主。
- 风险信息必须保留，不允许为了“温暖”而淡化严重问题。

## 六、验收标准

- 6 月底：完成至少 3 个老人画像和 5 组报告对比。
- 7 月底：`generate_family_report()` 可以独立生成老人版、护工版、家属版三类报告。
- 8 月底：参与 2-3 名外部试用者反馈整理，修正看不懂、不够准确或过度煽情的表达。
- 9 月：提供适合答辩展示的“传统版 vs CareMind 版”对比案例。

## 七、注意事项

- Agent 3 的优势是“有人味”，但前提是事实准确。
- 家属最关心的不是漂亮文字，而是老人是否安全、是否被认真照顾。
- 报告里可以出现生活细节，但必须来自输入数据或老人画像。
- 输出要体现 CareMind 的价值主张：让家属从“黑箱焦虑”转向“理解与信任”。

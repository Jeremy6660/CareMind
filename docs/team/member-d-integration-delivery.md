# 成员 D 分工安排：集成与交付负责人

## 一、团队总分工

CareMind 采用“四条主线、互相验收”的小组协作方式。每个人负责一个清晰模块，同时承担对其他成员产出的审查责任，避免最后集成时才发现接口、内容或安全边界不一致。

| 成员 | 角色定位 | 主责模块 | 核心交付物 | 评价指标 |
|---|---|---|---|---|
| A | 分诊评估负责人 | Agent 1 | 症状案例库、分诊 Prompt、分诊函数、测试记录 | 风险分级准确率、建议是否安全清晰 |
| B | 用药安全负责人 | Agent 2 | 药物风险 JSON、用药匹配逻辑、预警 Prompt、来源审核表 | 高风险组合不漏报、来源可追溯 |
| C | 家属沟通负责人 | Agent 3 | 老人画像、报告模板、沟通 Prompt、三方报告生成 | 表达温暖、信息准确、不掩盖风险 |
| D | 集成与交付负责人 | Orchestrator + Demo + 材料 | 项目骨架、Gradio Demo、端到端流程、报告/PPT 汇总 | Demo 可演示、接口统一、材料完整 |

互审机制：

- A 审 C：确认家属报告里的健康风险没有被“温暖表达”稀释。
- B 审 A：确认分诊建议没有越权诊断或医学风险。
- C 审 B：确认用药预警能被护工和家属读懂。
- D 审全员：确认接口、文件结构、演示流程可以集成。

## 二、D 的角色定位

D 是项目集成、Demo、报告与答辩负责人。D 不需要替 A/B/C 完成业务内容，但必须保证他们的成果能够合并成一个可运行、可展示、可讲清楚的系统。

D 的核心任务是“把项目收束起来”：统一目录、统一接口、统一演示流程、统一材料叙事。最终答辩时，评委看到的不是四份分散作业，而是一套完整的 CareMind 原型。

## 三、阶段任务

| 阶段 | 时间 | 任务重点 | 产出 |
|---|---|---|---|
| 准备期 | 6 月 | 搭建代码骨架、界面骨架和进度机制 | `code/` 目录、Gradio 空界面、报告背景初稿 |
| 开发期 | 7 月 | 协助 A/B/C 封装接口，统一输入输出 | Agent 接口规范、集成检查记录 |
| 集成期 | 8 月 | 开发 Orchestrator 和完整 Demo | `run_caremind()`、5 个端到端案例、演示视频 |
| 冲刺期 | 9 月 | 汇总报告、PPT、测试材料和答辩流程 | 最终报告、PPT、视频、备份包 |

## 四、具体措施

1. 建立项目代码目录结构，保持简单清晰。
2. 统一 A/B/C 三个 Agent 的输入输出格式，避免每个模块各说各话。
3. 建立 `code/agents/triage/`、`code/agents/medication_safety/`、`code/agents/family_communication/`、`code/orchestrator/`、`code/tests/` 等目录。
4. 开发 `run_caremind(elder_id, input_data) -> dict`，串联老人画像、分诊、用药安全和家属报告。
5. 搭建 Gradio 三步式 Demo：输入症状、显示分诊和用药预警、生成家属报告。
6. 每周维护一次进度表和问题清单，记录谁卡住、卡在哪里、下周如何解决。
7. 8 月录制 5-8 分钟演示视频，展示完整流程。
8. 9 月汇总研究报告、PPT、测试数据和答辩讲稿。

建议 Orchestrator 输出结构：

```json
{
  "elder_id": "elder_001",
  "triage_result": {
    "risk_level": "yellow",
    "action": "建议测量血压并观察。"
  },
  "medication_result": {
    "has_risk": true,
    "highest_risk_level": "high"
  },
  "family_report": {
    "family_version": "张奶奶本周精神不错，但 2026-06-08 出现轻微头晕..."
  },
  "system_notes": [
    "Agent 1 触发 Agent 2",
    "Agent 2 风险已写入家属报告"
  ]
}
```

建议 Demo 流程：

```text
步骤 1：选择老人档案，输入护工记录的症状和本周照护数据
步骤 2：展示 Agent 1 分诊结果和 Agent 2 用药安全预警
步骤 3：展示 Agent 3 生成的家属版、护工版、老人版报告
```

## 五、工作要求

- 不引入复杂技术栈，保持 `Python + JSON + Gradio`。
- 不提前使用 LangGraph、AutoGen、LlamaIndex、向量数据库或微信小程序框架。
- 不替 A/B/C 完成业务数据，但要及时发现他们的输出是否无法集成。
- 所有接口字段尽量稳定，避免每周大改。
- Demo 要优先保证流程清楚、结果可信，不追求复杂视觉效果。
- 报告和 PPT 要围绕“护工赋能、家属答疑、机构降本”组织叙事。
- 每周五检查一次本周产出是否已经提交到 Git。

## 六、验收标准

- 6 月底：代码骨架、Gradio 界面骨架、报告背景部分初稿完成。
- 7 月底：三个 Agent 都能以统一接口独立运行，Demo 至少能展示三个模块。
- 8 月底：5 个完整案例端到端跑通，Gradio Demo 可演示，演示视频完成。
- 9 月：最终报告、PPT、演示视频、测试结果和代码备份全部就绪。

## 七、注意事项

- D 是项目“收口人”，重点是让全组产出形成一个整体。
- 集成问题要尽早暴露，不要等到 8 月底才发现字段不统一。
- Demo 案例要精挑细选，建议使用“张奶奶头晕 + 用药风险 + 家属报告”作为主线。
- 答辩时不要陷入技术细节，要讲清楚养老场景、系统价值和安全边界。

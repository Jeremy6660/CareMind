# 成员 B 分工安排：Agent 2 用药安全负责人

## 一、团队总分工

CareMind 采用“四条主线、互相验收”的小组协作方式。每个人负责一个清晰模块，同时承担对其他成员产出的审查责任，避免最后集成时才发现接口、内容或安全边界不一致。

| 成员 | 角色定位 | 主责模块 | 核心交付物 | 评价指标 |
|---|---|---|---|---|
| A | 分诊评估负责人 | Agent 1 | 症状案例库、分诊 Prompt、分诊函数、测试记录 | 风险分级准确率、建议是否安全清晰 |
| B | 用药安全负责人 | Agent 2 | 药物风险 JSON、用药匹配逻辑、预警 Prompt、来源审核表 | 高风险组合不漏报、来源可追溯 |
| C | 家属沟通负责人 | Agent 3 | 老人画像、报告模板、沟通 Prompt、三方报告生成 | 表达温暖、信息准确、不掩盖风险 |
| D | 集成与交付负责人 | Orchestrator + Demo + 材料 | 项目骨架、Gradio Demo、端到端流程、报告/PPT 汇总 | Demo 可演示、接口统一、材料完整 |

互审机制：

- A 审 C：确认家属报告里的健康风险没有被“温暖表达”稀释。
- B 审 A：确认分诊建议没有越权诊断或医学风险。
- C 审 B：确认用药预警能被护工和家属读懂。
- D 审全员：确认接口、文件结构、演示流程可以集成。

## 二、B 的角色定位

B 是 Agent 2 用药安全负责人，负责建立药物风险知识库，并实现多药并用风险提示。

Agent 2 是整个项目中安全要求最高的模块。它不追求“看起来聪明”，而是追求来源可靠、规则可解释、输出谨慎。所有用药建议都必须明确提示由医生或药师复核，不能替代处方决策。

## 三、阶段任务

| 阶段 | 时间 | 任务重点 | 产出 |
|---|---|---|---|
| 准备期 | 6 月 | 整理药物风险库和预警 Prompt | `medication_conflicts.json`、来源记录、Prompt 初版 |
| 开发期 | 7 月 | 开发用药匹配和预警生成逻辑 | `check_medication_safety()`、测试药物组合 |
| 集成期 | 8 月 | 提供端到端演示中的典型用药风险案例 | 用药风险演示样例、误报/漏报记录 |
| 冲刺期 | 9 月 | 支持报告和答辩中“用药安全”部分 | 数据来源说明、典型风险图表 |

## 四、具体措施

1. 建立不少于 50 条药物风险记录，优先保证准确性，不追求大而全。
2. 每条记录至少包含：`drug_a`、`drug_b`、别名、风险等级、风险机制、可能后果、建议、来源、审核状态。
3. 优先覆盖抗凝药、NSAIDs、降压药、降糖药、镇静催眠药、心脏类药物。
4. 每条规则必须保留来源链接或来源名称，优先参考项目已有文档 `docs/medication-safety-knowledge-sources.md`。
5. 默认 `review_status` 设置为 `pharmacist_review_required`，表示正式使用前需要药学或医学背景人员复核。
6. 开发函数 `check_medication_safety(medications_list, patient_info) -> dict`。
7. 为每条高风险规则至少准备一个测试药物组合，验证不会漏报。

建议数据结构：

```json
{
  "id": "ddi_aspirin_ibuprofen_001",
  "drug_a": "阿司匹林",
  "drug_b": "布洛芬",
  "drug_a_aliases": ["aspirin"],
  "drug_b_aliases": ["ibuprofen"],
  "risk_level": "high",
  "risk_type": "bleeding_risk",
  "mechanism": "两者均可能增加胃肠道出血风险，老年人风险更高。",
  "effect": "可能增加消化道出血风险，合并抗凝药或既往出血史时需高度警惕。",
  "suggestion": "建议联系医生或药师复核，不要自行停药、减药或换药。",
  "source": [
    {
      "name": "AGS Beers Criteria 2023",
      "url": "https://pmc.ncbi.nlm.nih.gov/articles/PMC12478568/"
    }
  ],
  "review_status": "pharmacist_review_required"
}
```

建议输出结构：

```json
{
  "has_risk": true,
  "risk_count": 1,
  "matched_rules": ["ddi_aspirin_ibuprofen_001"],
  "highest_risk_level": "high",
  "caregiver_warning": "检测到可能存在用药风险，建议联系医生或药师复核。",
  "family_summary": "目前发现一种需要复核的用药组合，机构会持续记录并建议专业人员确认。",
  "safety_note": "请勿自行停药、减药或换药。"
}
```

## 五、工作要求

- 每条规则必须有来源链接或来源名称。
- 不允许直接使用无来源网络文章、论坛、短视频或科普截图作为入库依据。
- 输出必须提醒“联系医生/药师复核”。
- 不得建议老人或家属自行停药、换药、减药或加药。
- 对已录入的高风险组合，系统检测不能漏报。
- 对匹配不确定的药名，要保守提示人工复核，并记录原因。
- 用药风险文字必须让护工和家属读得懂，避免堆砌医学术语。

## 六、验收标准

- 6 月底：完成不少于 50 条初版药物风险规则，每条规则有来源和审核状态。
- 7 月底：`check_medication_safety()` 可以独立运行，对已录入高风险组合漏报率为 0。
- 8 月底：提供至少 1 个适合演示的典型用药风险案例，例如“阿司匹林 + 布洛芬”。
- 9 月：完成用药安全模块的数据来源说明、测试结果和答辩讲解素材。

## 七、注意事项

- Agent 2 是安全底线模块，宁可做少一点，也不能做错。
- 药物名称可能存在中文名、英文名、商品名、剂型差异，需要用别名字段降低漏报风险。
- 不能把系统包装成“药师替代品”，它只是风险提醒和沟通辅助工具。
- 答辩时要主动说明审核机制，这会增强可信度。
